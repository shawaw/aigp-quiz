BOK Competency I.A — Performance Indicators
  • I.A.1 Distinguish AI systems from traditional software systems and describe how AI systems learn and improve from data.
  • I.A.2 Identify and describe types of AI systems — supervised, unsupervised, reinforcement learning, and generative AI — and their typical use cases.
  • I.A.3 Describe the unique characteristics of AI that create governance challenges: opacity, emergent behaviour, scalability, adaptability, and dependency on data quality.
  • I.A.4 Identify key AI terminology: model, training data, inference, algorithm, foundation model, large language model (LLM), hallucination, bias.
  • I.A.5 Describe the general AI development lifecycle from problem framing through deployment and monitoring.
  • I.A.6 Explain why the unique characteristics of AI necessitate AI-specific governance beyond traditional IT governance.
Concepts Core Concepts & Definitions ▾
AI vs. Traditional Software

Exam frequently tests whether candidates can articulate why AI requires different governance — the answer always starts here.

Key distinctions
Traditional software executes explicit, human-written rules. Behaviour is deterministic and fully traceable to code.
AI systems derive behaviour from patterns in training data. Behaviour is probabilistic, can change with new data, and may not be fully explainable.
Governance implication — you cannot audit an AI system purely by reading its code. You must also audit its training data, model architecture, and outputs.
Types of AI Systems

Governance obligations differ by AI type. Exam scenarios often specify a type and ask what risk or control applies.

Supervised learning — trained on labelled data to predict outputs (e.g., fraud detection, credit scoring). Risk: label bias.
Unsupervised learning — finds patterns in unlabelled data (e.g., customer segmentation). Risk: opaque groupings that may encode discrimination.
Reinforcement learning — agent learns by reward/penalty from environment (e.g., recommendation engines). Risk: optimising a proxy metric that diverges from intended outcome.
Generative AI / Foundation models / LLMs — trained on massive corpora to generate content. Risks: hallucination, IP infringement, prompt injection, amplified bias at scale.
Unique Governance Challenges of AI

I.A.3 and I.A.6 are the highest-frequency PIs in Domain I. Every exam scenario about "why we need AI governance" maps to one of these five characteristics.

Opacity (black-box) — model reasoning cannot always be inspected. Blocks meaningful human review and contestability.
Emergent behaviour — capabilities appear at scale that were not designed or anticipated, making pre-deployment testing insufficient.
Scalability — a single model can make billions of decisions. Errors replicate at societal scale before they are detected.
Adaptability — models that retrain on new data can drift from validated behaviour without explicit deployment of new code.
Data dependency — output quality and fairness are determined by training data, not code. Garbage-in is invisible at the software layer.
Key Terminology

The AIGP exam uses IAPP/BOK terminology precisely. Using a term loosely will cost marks on scenario questions that hinge on definitional accuracy.

Model — the mathematical function produced by training an algorithm on data; it maps inputs to outputs at inference time.
Training data — the labelled or unlabelled dataset used to fit model parameters. Its quality and representativeness determine model fairness and accuracy.
Inference — the production use of a trained model to generate predictions or outputs on new, unseen data.
Hallucination — a generative AI output that is factually incorrect or fabricated, presented with apparent confidence. A key reliability and liability risk.
Bias — systematic error in model outputs caused by unrepresentative training data or flawed design choices. Distinct from statistical variance.
Foundation model — a large model trained on broad data at scale, adapted to many downstream tasks (e.g., GPT-4, Gemini, Claude). Governance challenge: a single upstream failure cascades across all downstream applications.
AI Development Lifecycle

The lifecycle is the scaffold on which Domains III and IV hang. Understanding it now lets you place every control you study in its correct phase.

1. Problem framing — define objective, determine if AI is the right tool, identify affected stakeholders.
2. Data collection & preparation — acquire, label, clean, and document training data. Primary locus of bias risk.
3. Model development & training — select algorithm, train, tune hyperparameters, document design choices.
4. Testing & evaluation — accuracy, fairness, robustness, and adversarial testing before release.
5. Deployment — release into production; governance gates, change management, rollback plans.
6. Monitoring & maintenance — ongoing performance, drift detection, incident response, periodic re-evaluation.

Scenario Worked Exam Scenario — Apply Level ▾
Situation
A regional bank's IT governance team proposes applying its existing software change-control policy to a new AI-driven loan decisioning system. The policy requires documented code review and sign-off before each release. The AI model retrains on new transaction data weekly without a new software release.
Your Role
AI Governance Professional advising the Chief Risk Officer. The CRO asks: is the existing policy sufficient?
Analysis
The existing policy fails on two dimensions. First, it targets code changes — but AI behaviour changes via data-driven retraining with no new code deployed I.A.3 — Adaptability. Second, code review cannot catch bias introduced through new training data; that requires data governance and fairness testing I.A.6 — AI-specific governance necessity. The model can become materially different between reviews without triggering any existing control gate.
Recommended Outcome
Supplement the change-control policy with: (1) a retraining governance procedure triggered by each weekly retrain cycle, not software releases; (2) automated fairness metric monitoring on inference outputs; (3) a human-in-the-loop escalation path for decisions above a lending threshold. Existing IT controls remain valid for infrastructure layers but cannot substitute for AI-specific controls at the model layer.
Recall Quick Recall — Click to Reveal ▾
? What single property of AI systems means you cannot audit them purely through code review?
Behaviour emerges from training data, not explicit code rules (I.A.1 / I.A.3). The data is the functional specification — code review sees only the algorithm shell.
? Name the five unique AI characteristics that create governance challenges per the BOK.
Opacity · Emergent behaviour · Scalability · Adaptability · Data dependency (I.A.3). Remember: OESAD.
? What is the governance risk specific to foundation models that does not apply equally to narrow AI?
Cascade risk — a single upstream model serves hundreds of downstream applications. A bias, error, or security vulnerability in the foundation model propagates instantly and at scale across all of them (I.A.4).
? Distinguish hallucination from bias in an LLM context.
Hallucination: factually incorrect output produced with apparent confidence — a reliability/accuracy issue. Bias: systematic skew in outputs toward or against particular groups — a fairness issue. Both can cause harm but require different controls (I.A.4).
? At which lifecycle phase does bias most commonly originate, and why?
Data collection and preparation (I.A.5). If the training dataset underrepresents or misrepresents a group, the model learns that skew. Downstream testing can detect it but not cure it without fixing the data.
? A model retrains on live data daily with no new code deployment. Which AI characteristic does this exemplify, and what control does it require?
Adaptability (I.A.3). Requires continuous monitoring and a retraining governance procedure that triggers review on each retrain cycle, independent of software release gates (I.A.6).
? What is the difference between training and inference in an AI lifecycle context?
Training: fitting model parameters to data — happens during development. Inference: applying the trained model to new inputs in production. Governance controls differ — training controls focus on data quality and fairness; inference controls focus on output monitoring and human oversight (I.A.4/I.A.5).
